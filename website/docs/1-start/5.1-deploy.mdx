## Requirements

On the project's side:

1. Fill in the necessary deployment info in `.aldev.yaml` - for instance replace the `???` placeholders with actual values
2. Force the regeneration of all the config files with `aldev confgen --regen --verbose ` or `aldev confgen -r -v`

On Azure's side:

- You have sufficient permissions / **activated roles** on Azure
- You're logged to Azure, with `az login`
- 1 AZ Storage Account & 1 Storage Container for the TF states (+ 1 Resource Group for both)
- 1 AZ Container Registry (+ 1 Resource Group)
- 1 AZ API Manager (APIM, + 1 Resource Group)
- Registering of the env subscritions:
  - `az provider register --namespace Microsoft.App --subscription sub-for-sandbox/staging/production`

## Creating (or updating) the GLOBAL resources

Run this to create an AZ app, and rights for Gitlab to use AZ for deployment purposes:

```sh
# From the project's root dir, go into the folder for the global infra elements
cd deploy/remote/a-infra/0-glo 

# Init the Terraform state
az account set --subscription "your-sub-where-your-apps-live" && terraform init

# Apply the main.tf file - don't forget to say "yes" :)
terraform apply
```

## Pushing a first release

Make sure you have locally [run](#running-the-new-api) the API a bit before moving on. **This created all the needed deployment config files along the way.**

When you're done testing - and maybe modifying / git syncing some stuff - do a first `SANDBOX` (dev) release by committing / pushing all the current changes, or should you want to try a `STAGING` release, do it with `aldev release`:

```sh
aldev release patch

# or, if you want to see what's going on
aldev release patch -v

# other options for `aldev release`: "minor", and "major"
```

<br/>

:::warning
The Gitlab pipeline should build successfully, but the deploy step should fail at this stage, because we yet have to create the AZ resources for the target environments, like for the [SANBOX env (dev)](#d-azure-deployment--gitlab-ci-sandbox-env-dev).
:::

## Creating (or updating) the SANDBOX env (e.g. "dev") resources

```sh
# From the project's root dir, go into the folder for the sandbox infra elements
cd deploy/remote/a-infra/1-dev 

# Init the Terraform state
az account set --subscription "your-sub-where-your-apps-live" && terraform init

# Apply the main.tf file - don't forget to say "yes" :)
terraform apply
```

If this fails, which might happen because the app is not deployed to ACA for `dev` yet (chicken & egg problem), then:

- re-do a [release](#pushing-a-first-release) or restart the latest whole Gitlab pipeline.
- to sync the TF state: `terraform apply`, which might fail again, then:
  - `terraform import module.infra_dev.azurerm_container_app.aca /subscriptions/xxxx/.../containerApps/aca-app-name`
  - `terraform plan` should say everything is fine now

Finally, to test your deployment, let's make an HTTP call, like this:

```sh
curl https://my-apim-name.azure-api.net/foobar-dev/rest/translation/fr\?Namespace\=Common 
```
